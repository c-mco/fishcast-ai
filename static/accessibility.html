FishCast AI aims to meet the Web Content Accessibility Guidelines 2.1 at conformance level AA. This page documents our current status, what we've fixed, and where known limitations exist. Last audited: 21 April 2026.

Current Status

We ran automated audits using axe-core 4 (via @axe-core/playwright) and 15 Playwright tests covering keyboard navigation, colour contrast, ARIA semantics, focus visibility, and reduced-motion. All 15 tests pass as of the last audit.

Known Limitations

The following issues are either inherent to third-party libraries or affect only enhanced conformance levels (AAA) that are not our current target.

Leaflet map — partial keyboard support

The interactive map is powered by Leaflet.js, which provides limited keyboard navigation. Zoom and pan with arrow/+/− keys work, but arbitrary panning to off-screen markers is not fully supported without a custom keyboard handler. This affects WCAG 2.1.1 (keyboard) in the map region specifically. All hotspot data is also accessible in the sidebar list as a text alternative — users who cannot use the map can interact entirely via the sidebar.

Plan: Evaluate leaflet-accessible or add a custom keyboard layer to cycle through markers with arrow keys.

Map popups — focus management

When a hotspot popup opens via keyboard, focus is not automatically moved into the popup. This affects WCAG 2.4.3 (focus order). Leaflet does not expose a native API for this; a custom popupopen event handler is needed to move focus.

Plan: Add map.on('popupopen', e => e.popup.getElement().focus()) and ensure the popup container has tabindex="-1".

Map tile layer — no text alternative for visual information

The ESRI Ocean Base map tiles are purely visual. Waterway names, coastline shapes, and depth indicators are not exposed as accessible text. This affects WCAG 1.1.1. The sidebar provides all functionally necessary information (hotspot names, species, distances) as plain text, which serves as a complete non-visual alternative.

Colour alone used for hotspot score grades

The score circle and spot list use colour (green → red) to indicate conditions grade. While the grade word ("Good", "Poor") is always present alongside the colour, WCAG 1.4.1 requires that colour is not the only means of conveying information. The grade text satisfies this, but the coloured dot on each spot item in the list has no text alternative.

Plan: Remove or aria-hidden the purely decorative dot; the grade is already in the aria-label of the button.

What Cannot Be Tested Automatically

Automated tools catch roughly 30–40% of WCAG issues. The following require human judgment or assistive technology testing that no automated tool can currently cover:

Screen reader experience

  • VoiceOver / NVDA reading order — automated tools check the DOM order but cannot verify that the reading experience feels logical and complete to a real screen reader user navigating the page for the first time.
  • Live region announcements — axe confirms aria-live exists, but only a screen reader user can confirm that loading step changes are announced at the right frequency (not too often, not missed).
  • Map landmark announcement — whether VoiceOver announces the map as "Interactive map showing fish hotspot observation hotspots…" and whether that's useful.

Cognitive accessibility — WCAG 2.2 / COGA

  • Plain language — terms like "research-grade iNaturalist observations", "conditions score", and "WCAG 2.1 AA" may be confusing to users unfamiliar with fishing or accessibility standards.
  • Error recovery — when GPS is denied or the API fails, error messages explain what happened, but whether the cause-and-recovery path is clear to all users requires human review.
  • Consistent navigation — the app is a single page, so navigation consistency (WCAG 3.2.3) cannot be fully tested with only one page.

Zoom and reflow — WCAG 1.4.10

  • The layout uses a fixed sidebar width (320px) beside a map. At 400% zoom (320px viewport equivalent), the sidebar and map overlap. A single-column reflow is not implemented. This is a known WCAG 1.4.10 failure for the current layout.
  • Plan: Add a responsive breakpoint that stacks the sidebar below the map at narrow viewport widths.

Touch and pointer — WCAG 2.5

  • Target size (WCAG 2.5.8 AA, 2.2) — hotspot list items are at least 44×44 CSS pixels. The accessibility link in the header (♿) is smaller; needs review.
  • Dragging — the map requires click-drag to pan. Leaflet provides pinch-zoom on touch. No alternative for single-pointer pan is provided (WCAG 2.5.7 AAA).

User-installed AT and browser extensions

  • Compatibility with browser-based assistive tools (e.g. Read Aloud, Helperbird, high-contrast browser extensions) has not been tested.
  • The dark colour scheme works with most high-contrast overlays, but forced-colours mode (Windows High Contrast) has not been tested.

How We Test

  • axe-core 4 via @axe-core/playwright — WCAG 2.1 AA automated scan on loading screen, full page, and sidebar in isolation after the app has fully initialised with real GPS and live data.
  • Playwright 15 dedicated a11y tests — keyboard Tab/Enter navigation, focus visibility, role=progressbar attributes, aria-live regions, emoji aria-hidden, score circle aria-label, source badge titles, and reduced-motion media query.
  • 44 functional Playwright tests — also run after every change to catch regressions in the app that could affect accessibility.
  • Manual review — HTML structure, landmark hierarchy, reading order, and this statement itself reviewed by a human.
